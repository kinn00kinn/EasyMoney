# 個人向け家計簿 Web アプリ 実装依頼書

## 目的

個人利用を前提とした家計簿 Web アプリを実装したい。
スマホとパソコンの両方から利用でき、**手入力中心**で記録する。
OCR は今回は対象外とする。
内部データは**複式簿記**で保持し、あとから分析しやすい構造にする。
将来的な拡張性・保守性を重視し、REST API ベースで設計する。

---

## ゴール

以下を満たす MVP（最小実用版）を構築したい。

- Web アプリとして動作する
- スマホ/PC 両対応の UI を持つ
- 家計簿を手入力できる
- 内部的には複式簿記で保存する
- 現金 / 銀行 / クレジットカードを扱える
- PayPay銀行の CSV を取り込める
- 基本的な分析グラフを表示できる
- 将来機能追加しやすい構造にする

---

## 前提条件

- 単一ユーザー利用を前提にする（マルチユーザーは後回し）
- 認証は最低限でよいが、セキュリティは考慮する
- クラウド基盤は **Cloudflare** を優先する
- CI/CD は **GitHub Actions** を利用する想定
- OCR は今回実装しない
- ネイティブアプリは作らず、**Web アプリのみ**で構築する

---

## 希望アーキテクチャ

以下をベースに提案・実装してほしい。

- **フロントエンド**: React + Vite
- **UI**: レスポンシブ対応（スマホ / PC）
- **バックエンド API**: Cloudflare Workers または Pages Functions
- **DB**: Cloudflare D1
- **将来のファイル保存**: 必要なら Cloudflare R2 に拡張可能な構成
- **デプロイ**: GitHub Actions 経由で Cloudflare へ自動デプロイ
- **API 形式**: REST API

フロントと API はできれば疎結合にし、あとで別クライアント（例: モバイルUI変更、外部連携）を追加しやすい構成にしてほしい。

---

## 機能要件

### 1. 取引の手入力

以下の情報を入力して保存できること。

- 日付
- 金額
- 内容（店名 / 摘要 / メモ）
- 支払方法
  - 現金
  - 銀行口座
  - クレジットカード

- カテゴリ
  - 例: 食費 / 日用品 / 交通費 / 娯楽 / 光熱費 など

- 任意メモ

#### 入力 UX 要件

- スマホで片手操作しやすいこと
- 金額入力を最優先でしやすい UI にすること
- よく使うカテゴリや支払方法を選びやすくすること
- 将来的に「前回入力の再利用」「候補補完」が追加しやすい構造にすること

---

### 2. 複式簿記での内部保存

UI 上は家計簿の単純入力に見せつつ、内部では**複式簿記の仕訳**として保存すること。

#### 例

- 現金で昼食 900円
  - 借方: 食費 900
  - 貸方: 現金 900

- クレジットカードで日用品 2,000円
  - 借方: 日用品費 2,000
  - 貸方: 未払金（クレジット）2,000

- 銀行口座から給与受取 250,000円
  - 借方: 普通預金 250,000
  - 貸方: 給与収入 250,000

#### 要件

- 1件の取引入力から、複数の仕訳行（entries）が生成されること
- 将来的に振替・調整仕訳・分割仕訳へ拡張しやすいこと
- 元帳の整合性（借方合計 = 貸方合計）を必ず担保すること

---

### 3. 口座管理

以下のような資産・負債の管理対象を持てること。

- 現金
- 銀行口座
- クレジットカード

#### 要件

- 口座一覧を持てる
- 口座ごとの残高を確認できる
- 口座の種別を判定できる
- 将来的に口座追加・無効化がしやすいこと

---

### 4. CSV 取り込み

**PayPay銀行の CSV** を初期対応対象として取り込み機能を実装したい。

#### 要件

- CSV をアップロードできる
- 行ごとにデータを読み込める
- 未確定の取引として取り込める
- 各行に対してカテゴリ付け・内容修正・確定ができる
- 重複取り込みをある程度防げること
- 将来的に銀行ごとの CSV フォーマット差異へ対応しやすい「インポートアダプタ構造」にすること

#### 補足

- 最初は PayPay銀行だけでよい
- ただし設計は他行追加しやすくしてほしい

---

### 5. 一覧・検索

記録した取引を見返しやすくしたい。

#### 要件

- 日付順に一覧表示
- 条件で絞り込み
  - 期間
  - カテゴリ
  - 支払方法
  - 口座

- 内容の簡易検索
- CSV 取込由来か手入力かが分かると望ましい

---

### 6. 分析・可視化

複数視点で分析しやすいグラフを表示したい。

#### MVP でほしいもの

- 月別の収支推移
- カテゴリ別の支出内訳
- 支払方法別の支出比率
- 口座残高推移

#### できれば入れたいもの

- **Sankey diagram**
  - 例: 収入 → カテゴリ → 支払手段
  - 例: 口座 → 支出カテゴリ

- 月次比較
- 期間指定分析

#### 要件

- 表示の元データは API 経由で取得できること
- あとで可視化種類を追加しやすいこと
- 集計ロジックを UI にべた書きしないこと

---

## UI / UX 要件

方向性として、**デジタル庁デザインシステムに近い、シンプルで見やすい UI** を目指したい。

### 基本方針

- 情報量は多すぎず、見通しがよい
- コントラストや余白を適切に取る
- スマホでも読みやすい
- フォーム入力が迷いにくい
- アクセシビリティに配慮する

### 画面構成の希望

#### スマホ

- 下部ナビゲーション
  - ホーム
  - 入力
  - 取込
  - 分析
  - 設定

#### PC

- 左サイドバーまたは上部ナビ

#### 最低限必要な画面

1. ダッシュボード
2. 取引入力画面
3. 取引一覧画面
4. CSV取込画面
5. 分析画面
6. 設定画面（口座・カテゴリ管理）

---

## 非機能要件

### 保守性

- 機能追加しやすい構成にする
- データモデルを明確に分離する
- API / UI / DB責務を分ける
- 命名をわかりやすくする
- 将来的なテスト追加がしやすいこと

### 拡張性

- REST API を整備し、フロントから API を呼ぶ構成にする
- 将来的に以下を追加しやすくする
  - OCR
  - 他銀行 CSV
  - 自動仕訳ルール
  - 外部連携
  - マルチユーザー化

### セキュリティ

- 単一ユーザー前提でも最低限の認証を考慮する
- 入力値バリデーション
- API の不正リクエスト耐性
- CSRF / XSS / インジェクション対策
- シークレットは Cloudflare / GitHub Secrets で管理
- ログに機微情報を過剰に出さない

### 性能

- 個人利用規模で十分軽快に動くこと
- 初期表示が重すぎないこと
- 集計 API が極端に遅くならないこと

---

## データモデル要件

最低限、以下のような構造を想定している。
必要に応じて調整・改善してよいが、**複式簿記の整合性を崩さないこと**。

### 想定テーブル

- `accounts`
  - 口座情報

- `categories`
  - 費目 / 収入カテゴリ

- `transactions`
  - 取引ヘッダ

- `entries`
  - 仕訳行（借方 / 貸方）

- `imports`
  - CSV 取り込み単位

- `imported_rows`
  - CSV の各行

- `rules`
  - 自動分類ルール（将来用でもよい）

### 設計方針

- `transactions` と `entries` を分ける
- 1つの取引に対して複数の仕訳行を持てる
- 借方/貸方の整合性を検証できるようにする
- 口座種別（資産 / 負債 / 収益 / 費用 / 純資産）を扱えるようにしておくと望ましい

---

## API 要件

REST API を実装してほしい。
エンドポイント名は調整可だが、概ね以下の責務を満たしてほしい。

### 例

- `GET /api/accounts`
- `POST /api/accounts`
- `GET /api/categories`
- `POST /api/categories`
- `GET /api/transactions`
- `POST /api/transactions`
- `GET /api/transactions/:id`
- `PATCH /api/transactions/:id`
- `POST /api/imports`
- `GET /api/imports/:id`
- `POST /api/imports/:id/confirm`
- `GET /api/analytics/summary`
- `GET /api/analytics/monthly`
- `GET /api/analytics/categories`
- `GET /api/analytics/sankey`

### API 設計方針

- 可能なら OpenAPI 的に整理しやすい形にする
- レスポンス形式を統一する
- バリデーションエラーを分かりやすく返す
- フロントが扱いやすい JSON 形式にする

## 実装で重視したいこと

以下の優先順位で実装してほしい。

1. **正しいデータ構造**
2. **シンプルで分かりやすい UI**
3. **将来の拡張性**
4. **CSV取込のしやすさ**
5. **分析画面の見やすさ**

見た目だけ先行するより、まずは**帳簿の整合性と保守性**を優先したい。

## 補足

- 「家計簿らしい使いやすさ」と「内部的な複式簿記」の両立を重視したい
- 初期はシンプルでよいが、あとで育てやすい土台にしたい
- 実装の都合で多少の設計変更提案があれば歓迎
- ただし、**単なるメモアプリではなく、帳簿として破綻しない構造**を守ってほしい

---

## 最終的にほしいもの

- 実装しやすい形に整理された設計
- MVP の動作する Web アプリ
- Cloudflare 上で動かしやすい構成
- 今後機能追加しやすいコードベース

以上の方針で、まずは設計から提案・実装を進めてください。
